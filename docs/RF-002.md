# RF-002 — Migración a backend real: autenticación, Firestore y monorepo

## 1. Información general

**Producto:** EricPay
**Tipo de iniciativa:** Migración de la Fase 0 (demo, SPEC-001) a la Fase 1 descrita en RF-001
**Depende de:** RF-001 (modelo de datos, reglas de negocio y stack ya definidos ahí no se repiten aquí salvo que cambien)
**Estado del código de partida:** Fase 0 implementada — app Expo de un solo proyecto, estado en memoria, login simulado (ver DECISIONS.md, incremento I-002)

## 2. Contexto

La Fase 0 (SPEC-001) demostró las tres modalidades de cobro por QR de principio a fin con estado en memoria y login simulado, sin backend. Ese código no se descarta: la UI, el sistema de diseño y la lógica de flujo se reutilizan. Lo que cambia es de dónde vienen los datos y quién es la fuente de verdad de los saldos.

Este RF cubre exactamente lo que SPEC-001 §6 dejó pendiente para "cuando la reunión valide el concepto": autenticación real, backend real, y la reestructuración a monorepo que ese backend justifica.

## 3. Objetivo

Reemplazar cada pieza simulada de la Fase 0 por su equivalente real, sin cambiar la experiencia de usuario ya validada:

1. Login simulado → Firebase Authentication con Google.
2. Store en memoria → Cloud Firestore como fuente de verdad.
3. Validaciones en cliente → Cloud Functions + Firestore Security Rules.
4. Proyecto Expo único → monorepo con `pnpm workspaces`.

## 4. Autenticación

Sustituir los botones "Continuar como Daniel/Laura" por Firebase Authentication con proveedor de Google, tal como especifica RF-001 §5. Primer acceso crea el perfil de usuario en Firestore (mismos campos que RF-001 §17.1) con saldo inicial configurable.

Los usuarios de demo (Daniel, Laura) dejan de estar precargados en código: se crean como cualquier otro usuario al autenticarse por primera vez.

## 5. Backend: Firestore

El modelo de datos de RF-001 §17 (usuario, cobro puntual, QR reutilizable, transacción) se implementa tal cual en Firestore. La capa de validación única que hoy vive en `src/domain/store.tsx` (Fase 0) se traduce a:

- **Cloud Functions** para las operaciones que mutan estado (crear cobro, pagar, crear/desactivar QR reutilizable). La ejecución del pago sigue siendo atómica (RF-001 §12) usando transacciones de Firestore.
- **Firestore Security Rules** para el control de acceso de lectura (RF-001 §16.5: cada usuario solo ve lo que le corresponde) y como segunda barrera contra escritura directa de saldos desde el cliente.

Las reglas de negocio en sí (RF-001 §18) no cambian — ya están implementadas y probadas en la Fase 0; este RF es de traducirlas de una capa de validación en cliente a una de backend, no de redefinirlas.

## 6. Tiempo real y persistencia

Los listeners de Firestore reemplazan el estado de React como mecanismo de actualización en vivo (RF-001 §13: el receptor ve su saldo actualizarse sin recargar). La sesión persiste entre aperturas de la app — a diferencia de la Fase 0, que arrancaba limpia en cada apertura.

## 7. Monorepo

Reestructurar a `pnpm workspaces` (RF-001 §16.3) separando:

- La aplicación móvil (código de Fase 0 migrado casi sin cambios: pantallas, tema, componentes).
- El backend y configuración de Firebase (Cloud Functions, Security Rules, Firestore indexes).
- Código compartido entre ambos, si emerge una necesidad real (tipos de dominio, por ejemplo) — no se crea un paquete `shared` de antemano sin un caso concreto que lo use.

La estructura exacta de carpetas se define durante la implementación, priorizando simplicidad sobre anticipar necesidades futuras.

## 8. Qué se reutiliza de la Fase 0 sin cambios

- Pantallas (`app/*`), sistema de diseño (`src/theme`, `src/components/ui.tsx`) y la forma de las tres modalidades de cobro.
- El payload de QR (`{app: 'ericpay', type, id}`) y la validación de que el QR pertenece a EricPay.
- Las reglas de negocio de RF-001 §18, ya verificadas manualmente en la Fase 0.

## 9. Qué se reemplaza

- `src/domain/store.tsx` (Context en memoria) → SDK de Firestore + llamadas a Cloud Functions.
- Login simulado (`app/login.tsx`) → flujo de Google Sign-In de Firebase Authentication.
- Usuarios de demo hardcodeados → altas reales por primer login.

## 10. Fuera de alcance

Igual que RF-001 §19: dinero real, integraciones bancarias, tarjetas, SEPA, criptomonedas, Apple/Google Pay, múltiples monedas, comisiones, KYC/AML, panel administrativo, notificaciones push, publicación en tiendas. Este RF no amplía el alcance funcional de RF-001, solo cambia dónde vive el estado.

## 11. Criterios de aceptación

1. Un usuario nuevo puede autenticarse con Google y EricPay le crea el perfil automáticamente.
2. Cerrar y reabrir la app mantiene la sesión iniciada (persistencia real).
3. Los tres escenarios de aceptación de RF-001 §20 (cobro puntual, QR personal, QR reutilizable) funcionan igual que en la Fase 0, ahora con Firestore como fuente de verdad y sin que el cliente pueda modificar saldos directamente.
4. Un intento de escritura directa de saldo desde el cliente es rechazado por las Security Rules.
5. Dos usuarios en dispositivos distintos ven la actualización de saldo e historial sin recargar manualmente.
